Business Summary

The core user authentication system has been successfully completed with no critical issues. However, the mobile app development team is currently blocked and cannot proceed until the technical details for this new authentication system are officially published by July 5th. We have established several key security measures and identified an immediate risk regarding password reset emails that requires attention.

Key Business Outcomes (4)
The new user login system is stable and ready for integration, providing reliable access for users.
The mobile app's launch timeline is at risk as the development team is blocked from integrating the login system until the authentication specification is published by July 5th.
Important security features have been implemented to protect user accounts, such as limiting password reset attempts to three per hour and session validity to 24 hours.
A critical security vulnerability exists because password reset emails are currently not limited, which could be exploited and needs to be addressed urgently.

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria (9)
Given a user successfully logs in, When a JWT token is issued, Then its validity period SHALL be 24 hours from the time of issuance.
Given a valid JWT token, When more than 24 hours have passed since its issuance, Then API requests using this token SHALL be rejected.
Given a user attempts to reset their password, When fewer than 3 password reset requests have been made by that user within the last 60 minutes, Then a password reset email SHALL be successfully sent.
Given a user attempts to reset their password, When 3 password reset requests have already been made by that user within the last 60 minutes, Then the password reset request SHALL be rejected, and no additional email SHALL be sent.
Given a user's password reset attempts have been rate-limited, When 60 minutes have elapsed since the first attempt in the rate-limited window, Then the user SHALL be able to request another password reset.
Given an authenticated user, When the user makes up to 100 API requests within a 60-second period, Then all requests SHALL be successfully processed.
Given an authenticated user, When the user attempts to make the 101st API request within a 60-second period, Then the request SHALL be rejected with a rate-limiting error.
Given an authenticated user's requests have been rate-limited, When 60 seconds have elapsed since the first request in the rate-limited window, Then the user SHALL be able to make subsequent API requests.
Given Sprint 3 is ready for sign-off, When checking the authentication module, Then automated integration tests for the login, refresh, and reset endpoints SHALL be implemented and passing.

Interface Contracts (4)

JWT tokens issued by the Authentication Module SHALL include an expiry claim ('exp') set for 24 hours from issuance.
The password reset API endpoint SHALL enforce a server-side rate limit of 3 attempts per hour per user, identifiable by a unique user identifier (e.g., IP address, user ID, email address).
All API endpoints exposed via AWS API Gateway SHALL be subject to a global rate limit of 100 requests per minute per user, based on the requesting user's identity.
