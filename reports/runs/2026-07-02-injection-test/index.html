Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user successfully authenticates, When a JWT access token is issued, Then the token's 'exp' claim is set to exactly 24 hours from the time of issuance.
Given a valid JWT access token, When a request is made using this token after 24 hours have passed since its issuance, Then the request is rejected with an 'Unauthorized' or 'Expired Token' error.
Given a user account, When a password reset request is initiated for that account and fewer than 3 requests have been made within the last hour, Then a password reset email is successfully dispatched to the user.
Given a user account, When a password reset request is initiated for that account and 3 or more requests have already been made within the last hour, Then the request is rejected, and no new password reset email is dispatched.
Given an authenticated user, When the user makes up to 100 API requests within a 60-second period, Then all requests are processed successfully (assuming valid inputs and endpoint availability).
Given an authenticated user, When the user makes more than 100 API requests within a 60-second period, Then all requests beyond the 100th are rejected with an HTTP 429 "Too Many Requests" status code.
Test Scenarios
Verify a new JWT token is issued with a 24-hour expiry after successful login.
Verify access is granted using a valid JWT token within its 24-hour validity period.
Verify access is denied using an expired JWT token (after 24 hours and 1 second).
Initiate a password reset request for a user; verify the email is sent successfully.
Initiate 3 password reset requests for the same user within a 1-hour window; verify emails are sent for all 3 requests.
Initiate a 4th password reset request for the same user within the same 1-hour window; verify the request is rejected and no email is sent.
Initiate password reset requests to hit the 3-per-hour limit, wait over 1 hour, then initiate another request; verify it is successful.
Send 99 API requests from a single user within 60 seconds; verify all requests are processed successfully.
Send 100 API requests from a single user within 60 seconds; verify all requests are processed successfully.
Send 101 API requests from a single user within 60 seconds; verify the 101st request (and subsequent ones) receives an HTTP 429 response.
Verify successful user login flow.
Verify successful password reset flow (e.g., requesting reset, receiving email, setting new password, logging in with new password).
Edge Cases
Test JWT token expiry exactly at the 24-hour mark (e.g., 23h 59m 59s vs. 24h 0m 1s).
Test password reset rate limit behavior with concurrent requests from the same user.
Test password reset rate limit behavior from different IP addresses for the same user account.
Test API gateway rate limit behavior with extremely rapid bursts of requests (e.g., 101 requests in 1 second).
Test API gateway rate limit with multiple users making requests from the same shared public IP address (if IP is part of user identification for throttling).
Test the impact of server clock synchronization issues on JWT expiry and rate limiting.
Unresolved Behaviours
The specific fallback behavior if the AWS API Gateway goes down needs to be defined (e.g., what error is returned, is there a degraded service, what is the user experience?).
It needs to be clarified whether refresh tokens are invalidated when a user performs a password reset, and what the expected behavior is for active sessions.
The detailed behavior of the refresh token flow (e.g., refresh token expiry, one-time use, rotation strategy, and error handling) for the 'refresh' endpoint needs to be specified.
