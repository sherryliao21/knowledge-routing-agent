Business Summary

This briefing covers key decisions made regarding our system's security, user authentication, and API development. Important steps are being taken to protect user accounts and ensure system stability. A critical dependency for the mobile app team's progress has also been identified and scheduled.

Key Business Outcomes
User accounts will be better protected from malicious attacks due to new limits on password reset attempts and overall system usage.
The mobile app development timeline is directly impacted; work is blocked until a key technical specification for user authentication is finalized and published by July 5th.
The reliability and quality of our user login and password reset functions will improve significantly with new automated testing requirements, ensuring a smoother user experience.
Our system will have enhanced stability and performance due to the implementation of strict limits on how many requests can be made at one time.

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user successfully authenticates and receives a JWT, When 23 hours and 59 minutes pass since the token was issued, Then the JWT token must still be valid for authentication.
Given a user successfully authenticates and receives a JWT, When 24 hours and 1 minute pass since the token was issued, Then the JWT token must be expired and invalid for authentication.
Given a user attempts to reset their password, When they have initiated fewer than 3 password reset requests within the last hour, Then a password reset email must be successfully sent to the user.
Given a user attempts to reset their password, When they have initiated 3 or more password reset requests within the last hour, Then no password reset email must be sent to the user, and the system must provide an appropriate message indicating the rate limit has been reached.
Given an authenticated user sends requests to the API, When the user sends 100 or fewer requests within a 60-second period, Then all requests must be processed successfully by the API gateway.
Given an authenticated user sends requests to the API, When the user sends more than 100 requests within a 60-second period, Then all subsequent requests exceeding 100 within that 60-second period must be rejected with a '429 Too Many Requests' HTTP status code (or equivalent).
Automated integration tests must be implemented for the 'login' authentication endpoint.
Automated integration tests must be implemented for the 'refresh token' authentication endpoint.
Automated integration tests must be implemented for the 'password reset' authentication endpoint.
All automated integration tests for the authentication endpoints must pass before Sprint 3 sign-off.
Test Scenarios
Successful login: Verify a valid JWT is returned and can be used immediately.
Near JWT expiry check: Use a valid JWT to make an API call at 23 hours 59 minutes after issuance.
Post-JWT expiry check: Attempt to use the same JWT to make an API call at 24 hours 1 minute after issuance, expect rejection.
Password reset successful request: Request a password reset for a valid user. Verify email is sent.
Password reset boundary success: Request password reset 3 times within a 30-minute window for the same user. Verify 3 emails are sent.
Password reset rate limit trigger: Request password reset a 4th time within the same 30-minute window for the same user. Verify no email is sent and an appropriate error message (e.g., 'Too many reset attempts, please try again later') is displayed.
Password reset rate limit reset: Wait 61 minutes after the 4th failed attempt, then request another password reset successfully for the same user. Verify email is sent.
API Gateway rate limit within limit (99 requests): Send 99 requests to a user-authenticated endpoint within a 50-second window. Verify all succeed.
API Gateway rate limit at limit (100 requests): Send 100 requests to a user-authenticated endpoint within a 50-second window. Verify all succeed.
API Gateway rate limit exceed limit (101st request): Send 101 requests to a user-authenticated endpoint within a 50-second window. Verify the 101st request receives a 429 status code.
API Gateway rate limit recovery: After receiving a 429, wait 61 seconds and then send another request. Verify it succeeds.
Auth API integration tests existence: Confirm the presence and execution of automated integration tests for login, refresh, and password reset endpoints.
Auth API integration tests passing: Verify that these tests pass as part of the CI/CD pipeline or before Sprint 3 sign-off.
Edge Cases
JWT Token: Attempt to use a valid token from a significantly different system clock time (e.g., client clock is off).
JWT Token: Token validation when network latency causes requests to arrive just at or after the 24-hour mark.
Password Reset: Attempt multiple password resets for a non-existent user. Verify no email is sent and a generic message is displayed (to prevent user enumeration).
Password Reset: Attempt password resets for the same user from different IP addresses within an hour. (Expected: Should still hit the limit for the user).
Password Reset: Attempt password resets for different users from the same IP address within an hour. (Expected: Should not block other users).
Password Reset: Rapid-fire requests for password reset (e.g., 5 requests in 1 second).
API Gateway Rate Limiting: Concurrent requests from a single user that cross the rate limit threshold (e.g., 50 requests, then 51 more concurrently).
API Gateway Rate Limiting: Requests from different authenticated users sharing the same public IP address (if applicable to 'per user' definition).
API Gateway Rate Limiting: Verify that API Gateway rate limiting applies to all authentication endpoints as well as other user-authenticated API endpoints.
