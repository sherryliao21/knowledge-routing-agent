Stakeholder Executive Briefing

Business-focused summary: impact, alignment decisions, and top open questions.

Business Summary

The foundational user authentication system has been delivered successfully with no critical issues. However, our mobile app development is currently stalled because the full technical specifications for the authentication system are not yet published, although a target date of July 5th has been set. There's also a notable security concern regarding unrestricted password reset emails, which needs immediate attention.

Key Business Outcomes
The core user login experience is stable and delivered to a high quality standard.
Mobile app development timelines are directly impacted and delayed until the authentication details are finalized and shared.
There is a significant security vulnerability related to password resets that could be exploited if not addressed.
User security might be compromised if old login sessions remain active after a password reset.

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user successfully obtains a JWT token, when 24 hours have elapsed since the token was issued, then subsequent API requests using that token must be rejected as unauthorized.
Given a user successfully obtains a JWT token, when less than 24 hours have elapsed since the token was issued, then subsequent API requests using that token must be successfully authorized.
Given a user requests a password reset for a specific email address, when 3 password reset attempts have already been made for that email within the last hour, then no further password reset email is sent for that email within the current hour, and the system provides an appropriate user message indicating the limit has been reached.
Given a user requests a password reset for a specific email address, when fewer than 3 password reset attempts have been made for that email within the last hour, then a password reset email is successfully sent to the user.
Given an authenticated user, when the user sends 100 API requests within a 60-second period, then all 100 requests are successfully processed.
Given an authenticated user, when the user sends 101 API requests within a 60-second period, then the 101st request is rejected with an HTTP 429 Too Many Requests status code.
Test Scenarios
Successfully log in, receive a JWT token, and verify it works immediately. Wait 24 hours and 1 minute, then try to use the same token and verify authentication fails.
Successfully log in, receive a JWT token, and verify it works immediately. Wait 23 hours and 59 minutes, then try to use the same token and verify authentication still works.
Initiate password reset 3 times for the same email address within a 5-minute window and verify that the system indicates 3 attempts were successful (e.g., 3 emails sent or 3 success messages).
Initiate password reset a 4th time for the same email address within the same 5-minute window and verify no email is sent and a rate limit error/message is displayed to the user.
Initiate password reset for an email, wait 61 minutes, then initiate again and verify an email is successfully sent.
Send exactly 100 API requests as a single authenticated user within 50 seconds and confirm all requests return successful responses.
Send 101 API requests as a single authenticated user within 50 seconds and confirm the 101st request returns an HTTP 429 status code.
Send 100 API requests as User A and 100 API requests as User B concurrently within 50 seconds, verifying all 200 requests succeed without rate limiting User A or User B.
Edge Cases
For password reset rate limiting: What if the email address provided for password reset does not exist? Does it count towards the 3 attempts per hour limit for that non-existent email to prevent enumeration?
For password reset rate limiting: What if multiple password reset attempts are made concurrently for the same user?
For API Gateway rate limiting: What specific mechanism identifies a 'user' for the 100 requests per minute limit? Is it based on an authenticated session ID, IP address, or something else?
Unresolved Behaviours
When a user's password is reset, should their existing refresh tokens be automatically invalidated?
