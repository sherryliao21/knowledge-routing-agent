Stakeholder Executive Briefing

Business-focused summary: impact, alignment decisions, and top open questions.

Business Summary

The core user authentication system was successfully launched last week with no critical issues, marking a significant step forward. However, the mobile app team is currently on hold, awaiting the final details for how to connect to this new system, which is targeted for release by July 5th. We also need to address a potential security vulnerability related to password reset emails not being properly limited.

Key Business Outcomes
Users will experience a robust and secure login process, with safeguards like limited password reset attempts and regular session refreshes.
There is a current delay for the mobile app team, impacting their ability to integrate the new authentication system until its technical specifications are published.
There's a critical security risk related to how password reset emails are handled, which could leave user accounts vulnerable if not addressed immediately.
The overall stability and security of our API interactions are enhanced through a dedicated gateway with usage limits, protecting against abuse.

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user logs in successfully and receives a JWT, When 24 hours pass from the JWT issuance time, Then attempts to use the JWT for authenticated requests must fail.
Given a user initiates a password reset, When the user makes more than 3 password reset attempts within a 60-minute window, Then the 4th password reset attempt within the same 60-minute window must be rejected.
Given a user sends requests through the API Gateway, When the user sends more than 100 requests within a 1-minute window, Then subsequent requests from that user within the same 1-minute window must be rejected.
The Login API endpoint must successfully authenticate a user with valid credentials and return a valid JWT.
The Refresh API endpoint must successfully issue a new access token when provided with a valid refresh token.
The Reset API endpoint must successfully process a password reset request and allow the user to set a new password.
Test Scenarios
Verify a user can make authenticated requests immediately after login using a valid JWT.
Verify authenticated requests fail when attempted with a JWT that has expired (after 24 hours and 1 minute).
Verify a user can make 3 password reset attempts within a 30-minute period and all are processed.
Verify a user's 4th password reset attempt within a 60-minute period is rejected.
Verify a user can successfully make a password reset attempt after waiting 1 hour and 1 minute following 3 previous attempts.
Verify a user can send 50 requests through the API Gateway within a 30-second period and all are successful.
Verify a user's 101st request within a 1-minute period through the API Gateway is rejected.
Verify a user can send 100 requests in 1 minute, wait 1 minute, then send another 100 requests successfully.
Verify a user can successfully log in with valid credentials.
Verify login fails with invalid credentials (e.g., incorrect password, non-existent user).
Verify an access token can be successfully refreshed using a valid refresh token.
Verify an access token refresh fails with an invalid or expired refresh token.
Verify the full password reset flow can be successfully completed by a user.
Edge Cases
Test JWT behavior precisely at the 24-hour expiry mark (e.g., 24 hours 0 minutes 0 seconds vs 24 hours 0 minutes 1 second).
Test multiple concurrent password reset attempts from the same user account.
Test invalid password reset attempts (e.g., wrong username or email) contributing to the rate limit.
Test sending exactly 100 requests through the API Gateway within 59 seconds.
Test sending 101 requests as close to simultaneously as possible through the API Gateway within 1 minute.
Test login attempts with empty or extremely long username/password inputs.
Test login attempts with special characters in username/password fields.
Unresolved Behaviours
Clarification is needed on whether password reset emails should also be rate-limited, separate from the password reset attempt limit (D004 vs D003).
Clarification is needed on whether existing refresh tokens are invalidated when a user successfully resets their password (D011).
