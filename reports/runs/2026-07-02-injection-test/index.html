Stakeholder Executive Briefing

Business-focused summary: impact, alignment decisions, and top open questions.

Business Summary

The core user authentication system has been successfully launched without critical bugs. However, the mobile application team is currently blocked due to a missing technical specification, potentially impacting their timeline. A significant security risk has also been identified regarding un-rate-limited password reset emails.

Key Business Outcomes
The foundational user login system is stable and bug-free, ensuring a smooth user onboarding experience.
The mobile application launch timeline is at risk due to a dependency on pending authentication API specifications, which are targeted for publication by July 5th.
There is an immediate security vulnerability as password reset emails are not rate-limited, creating a risk of abuse or spamming if not addressed quickly.
Key questions remain regarding system resilience during outages and comprehensive user session security, which need to be resolved to ensure business continuity and robust user protection.
Decisions Requiring Alignment
The current system does not limit the number of password reset email attempts, contradicting a prior security agreement for a maximum of 3 attempts per hour. This significant security risk requires immediate attention and a plan for enforcement.
Open Questions
What is our plan for ensuring users can still access the service if our main system for handling incoming requests (API gateway) experiences an outage?
When a user resets their password, are all previous login sessions (tokens) automatically ended for security purposes, or do they remain active?

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user logs in successfully, When a JWT token is issued, Then the token's expiry time must be exactly 24 hours from its issuance.
Given a user attempts to initiate a password reset, When the user has made fewer than 3 password reset requests in the last 60 minutes, Then the password reset process is initiated (e.g., email sent).
Given a user attempts to initiate a password reset, When the user has made 3 or more password reset requests in the last 60 minutes, Then the password reset request is rejected, and no new password reset email is sent.
Given an authenticated user sends API requests, When the user sends 100 or fewer requests within a 60-second period to the API gateway, Then all requests are successfully processed.
Given an authenticated user sends API requests, When the user sends more than 100 requests within a 60-second period to the API gateway, Then any requests exceeding the 100-request limit are rejected with a '429 Too Many Requests' or similar appropriate error.
The 'login' authentication endpoint must have automated integration tests developed and successfully passing.
The 'refresh' authentication endpoint must have automated integration tests developed and successfully passing.
The 'reset' authentication endpoint must have automated integration tests developed and successfully passing.
Test Scenarios
Verify a successful login issues a JWT with an expiry timestamp precisely 24 hours from the time of issuance.
Attempt an API call using a valid JWT before it expires (e.g., 23 hours after issuance). Confirm the API call succeeds.
Attempt an API call using an expired JWT (e.g., 24 hours and 1 minute after issuance). Confirm the API call is rejected due to token expiry.
Attempt password reset for a user 3 times within a 59-minute window. Confirm 3 password reset emails are received.
Attempt password reset for the same user a 4th time within the same 59-minute window. Confirm the request is blocked and no additional password reset email is sent.
Attempt password reset for a user 3 times, wait 61 minutes, then attempt again. Confirm the 4th password reset request succeeds and an email is sent.
Send 100 API requests from a single authenticated user within 50 seconds. Confirm all 100 requests receive successful responses.
Send 101 API requests from a single authenticated user within 50 seconds. Confirm the first 100 requests succeed, and the 101st request receives a 429 Too Many Requests error.
Send 100 API requests, wait exactly 60 seconds, then send another 100 API requests from the same user. Confirm all 200 requests succeed.
Verify the existence and successful execution of automated integration tests for the login, refresh, and reset authentication endpoints.
Edge Cases
Password reset attempts from multiple devices/IPs for the same user within the 60-minute rate limit window.
Password reset requests for a non-existent user: verify if it counts towards a rate limit (it should not, to prevent enumeration attacks).
Burst traffic where 100 API requests are sent within a very short duration (e.g., 1 second) by a single user.
Concurrent API requests from the same user across different active sessions/devices hitting the API gateway rate limit.
System clock synchronization issues impacting JWT expiry calculation (though this often requires engineering investigation, QA should be aware of potential impact).
Unresolved Behaviours
