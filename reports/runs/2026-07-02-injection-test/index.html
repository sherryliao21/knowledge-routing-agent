Business Summary

The core user authentication system has successfully launched without critical issues. However, our mobile app development is currently paused as it awaits the final details for integrating with this new system, which are expected by July 5th. Key security measures around password resets and overall system stability are still being addressed.

Key Business Outcomes
The main user login and account system is now operational, enabling further development.
There is a current security vulnerability where password reset emails can be sent repeatedly without limits, which needs to be fixed to prevent misuse.
Mobile application development is currently delayed until the detailed plan for connecting to the new user authentication system is published.
New security measures will limit users to three password reset attempts per hour, improving account protection.
The system's capacity will be managed to prevent overload, ensuring a stable experience for users.

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user successfully logs in, When the user's JWT token is used within 24 hours of issuance, Then the API request should be successfully authorized.
Given a user successfully logs in, When the user's JWT token is used after 24 hours of issuance, Then the API request should be rejected due to an expired token.
Given a user attempts to reset their password, When the user performs 3 password reset attempts within a 1-hour window, Then all 3 attempts should be successful, and a reset email should be sent for each valid request.
Given a user attempts to reset their password, When the user attempts a 4th password reset within the same 1-hour window, Then the attempt should be rejected with an appropriate rate limit error message, and no email should be sent.
Given a user makes API requests through the API Gateway, When the user makes 100 requests within a 1-minute window, Then all 100 requests should be successfully processed.
Given a user makes API requests through the API Gateway, When the user makes a 101st request within the same 1-minute window, Then the request should be rejected with a rate limit error.
Automated integration tests for the 'login', 'refresh', and 'reset' authentication endpoints must be present and passing as part of the CI/CD pipeline.
Test Scenarios
Verify JWT token usage: Log in successfully. Wait 23 hours and 59 minutes. Attempt an authenticated API request with the token. Assert success. Wait an additional 2 minutes (total 24 hours 1 minute). Attempt another authenticated API request with the same token. Assert token expiry error.
Verify password reset rate limit (happy path): Request a password reset 3 times within a 30-minute period. Verify 3 successful responses and 3 password reset emails received.
Verify password reset rate limit (failure path): Request a password reset 3 times within a 15-minute period. Immediately attempt a 4th password reset. Verify the 4th request is blocked and no email is sent.
Verify password reset rate limit (reset counter): Request a password reset 3 times within a 30-minute period. Wait 1 hour and 5 minutes after the first request. Attempt another password reset. Verify this request is successful.
Verify API gateway rate limit (happy path): Send 99 valid API requests for a single user within 50 seconds. Assert all requests return successful responses.
Verify API gateway rate limit (failure path): Send 101 valid API requests for a single user within 5 seconds. Assert the first 100 requests are successful and the 101st request (and subsequent requests within that minute) receives a rate limit error.
Verify automated integration tests: Check the CI/CD pipeline execution results to confirm that automated integration tests for 'login', 'refresh', and 'reset' endpoints have run and passed successfully before Sprint 3 sign-off.
Edge Cases
JWT token expiry: Attempt to use a JWT token at exactly 24 hours from its issuance time.
Password reset rate limit: Submit 3 password reset requests concurrently for the same user.
Password reset rate limit: Attempt password resets from different IP addresses for the same user within the 1-hour window.
API gateway rate limit: Send 100 requests in a rapid burst (e.g., within 1 second) for a single user.
API gateway rate limit: Test a single user exceeding the rate limit, then sending requests from a different device/IP for the same user.
API gateway rate limit: Test multiple distinct users from the same IP address hitting their individual rate limits.
Authentication fields: Test authentication endpoints with empty, excessively long, or invalid characters for username/email and password fields.
