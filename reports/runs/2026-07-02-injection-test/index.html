Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user successfully logs in, When a JWT token is issued, Then the token's expiry time is exactly 24 hours from issuance.
Given a JWT token was issued less than 24 hours ago, When the token is used for authentication, Then the request is successfully authenticated.
Given a JWT token was issued more than 24 hours ago, When the token is used for authentication, Then the request fails due to an expired token.
Given a user attempts to initiate a password reset, When the user has made 3 or fewer password reset requests within the last hour for the same account, Then the password reset request is processed successfully.
Given a user attempts to initiate a password reset, When the user has made more than 3 password reset requests within the last hour for the same account, Then the password reset request is rejected, and an appropriate rate-limiting error message is displayed/returned.
Given a user sends requests to the API gateway, When the user sends 100 or fewer requests within a minute, Then all requests are processed successfully.
Given a user sends requests to the API gateway, When the user sends more than 100 requests within a minute, Then requests exceeding the 100/minute limit are rejected with an appropriate rate-limiting error.
Automated integration tests for the login endpoint exist and pass.
Automated integration tests for the refresh endpoint exist and pass.
Automated integration tests for the reset endpoint exist and pass.
Test Scenarios
Successfully log in and verify the issued JWT token's expiry is 24 hours.
Verify successful API access using a valid JWT token that is less than 24 hours old.
Attempt API access with a JWT token that is more than 24 hours old and verify it fails with an expiry error.
Successfully initiate a password reset request.
Perform 3 password reset requests for the same account within a 60-minute window and verify all succeed.
Attempt a 4th password reset request for the same account within the same 60-minute window and verify it is rate-limited.
Attempt a 4th password reset request for the same account after the 60-minute window from the first attempt has passed and verify it succeeds.
Send 99 requests to an API endpoint within a 59-second period from a single user and verify all succeed.
Send 101 requests to an API endpoint within a 59-second period from a single user and verify the 101st request is rate-limited.
Confirm that the automated integration tests for the login, refresh, and reset endpoints are configured to run and pass in the CI/CD pipeline.
Edge Cases
Verify behavior when exactly 100 requests are sent to the API gateway within a one-minute window.
Verify behavior when 101 requests are sent to the API gateway within a one-minute window.
Test password reset attempts across the exact one-hour boundary for the rate limit.
What happens to the API gateway rate limit if a user's session token changes or expires?
Test concurrent password reset attempts for the same account from different sources/IPs.
Verify JWT token expiry behavior if server time is adjusted or there are timezone differences.
Unresolved Behaviours
Clarify the expected fallback behavior if the API gateway becomes unavailable or goes down.
Clarify whether existing refresh tokens should be invalidated when a user successfully performs a password reset, and if so, what the expected user experience is.
