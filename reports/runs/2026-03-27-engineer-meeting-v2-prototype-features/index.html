Stakeholder Executive Briefing

Business-focused summary: impact, alignment decisions, and top open questions.

Business Summary

The team has finalized key decisions for the first version of the new system, focusing on essential photo upload and data modeling capabilities. Users will log in securely using our existing Single Sign-On system. The initial release will provide a straightforward 'Normal Mode' for processing, with advanced options planned for future updates.

Key Business Outcomes (5)
Users will have a seamless and secure login experience using our existing company login system, reducing new password fatigue.
The first version will deliver core value by allowing users to upload photos and generate data models, enabling initial adoption and feedback.
The system will clearly organize user content, separating raw photos into 'Albums' and processed data into 'Projects'.
Users will see clear progress updates during data modeling, improving transparency and managing expectations.
The initial version will be user-friendly with a 'Normal Mode' for photo processing, making it accessible to a wider audience, with more advanced features to come.

Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria (12)
Given a user attempts to log in, When the user is redirected to the external SSO Portal for authentication, Then the user is successfully logged in upon successful SSO authentication.
Given a user accesses the system, Then no internal login or registration forms are displayed.
Given a user is logged in, When the sidebar is visible, Then distinct sections labeled 'Album' and 'Project' are displayed.
Given a user selects an 'Album' folder, When new photos are presented for upload, Then the system allows photos to be added to the selected 'Album'.
Given an 'Album' contains photos, When the user selects a photo and initiates a delete action, Then the selected photo is removed from the 'Album'.
Given an 'Album' contains photos, When the user enters a search query in the photo search bar, Then only photos matching the query are displayed within the 'Album'.
Given a modeling process is initiated, When the process is active, Then a progress bar is visibly displayed.
Given a modeling process is active and a progress bar is displayed, When the process moves through stages (e.g., AT computing, point cloud generation, model generation), Then the progress bar visually updates to indicate the status of each stage.
Given a user is creating a new 'Project', When prompted to import photos, Then the system displays a list of available photos for selection.
Given a user has selected photos from a list during 'Project' creation, When the user confirms the selection, Then the selected photos are imported into the new 'Project' folder.
Given a photo processing task is initiated, Then the system uses 'Normal Mode' parameters by default.
Given a photo processing task is initiated, Then no options for 'Expert Mode' parameter adjustment are presented.

View Full Decision History (8)
It was decided that user authentication will exclusively rely on an external SSO system via the Portal (D001), and the system will not implement its own login or registration functionality (D002). This decision simplifies the authentication stack by leveraging existing infrastructure.
Photo upload (D003) and Output modeling (D004) were confirmed as the two main features to prioritize for the V1 implementation, setting the core scope for the initial release.
The user interface will clearly differentiate between two folder types in the sidebar: 'Album' for photos and 'Project' for modeling data (D005).
The 'Album' folder type must support core photo management functionalities including adding (D006), deleting (D007), and searching photos (D008).
Modeling progress will be visualized using a progress bar (D009) that displays the status of each specific stage, such as AT computing, point cloud generation, and model generation (D010).
When creating a 'Project', users will have the ability to select and import photos from an existing photo list into the project folder (D011).
A clear scope distinction was made for future versions, deferring features such as Project sharing (D012), ensuring Albums are private to the owner (D013), resumable uploads (D014), display of storage quota (D015), dynamically adjustable photo-upload limits (D016), AT error handling with parameter adjustment (D017), and the distinction between 'Normal Mode' and 'Expert Mode' (D018).
Specifically for photo processing, V1 will only implement 'Normal Mode' (D019), aligning with the broader decision to defer more complex features to later releases.

Data Flows & Integrations (7)
User authentication requests will be redirected from This System to the External Portal, and authentication responses (e.g., tokens, user identity) will flow back from the External Portal to This System for user authentication. (Inferred from D001)
Client application transmits photo data to This System for storage within 'Album' folders (D006).
This System processes requests to store, retrieve, delete, and search photo metadata and binary data within 'Album' folders (Inferred from D006, D007, D008).
This System transfers references or copies of photos from 'Album' storage to 'Project' storage/data structures upon user selection (D011).
Photos and parameters from 'Project' folders are sent from This System to an external modeling computation service for processing (Inferred from D004, D019).
The external modeling computation service transmits real-time progress and stage status (e.g., AT computing, point cloud generation, model generation) back to This System for display (D009, D010).
The external modeling computation service returns generated modeling data (e.g., 3D models, point clouds) to This System for storage and display within 'Project' folders (Inferred from D004).

SSO Integration Contract: This System MUST integrate with the external Portal's SSO service. Specifics (e.g., OAuth 2.0 / OpenID Connect 'authorize' and 'token' endpoints, SAML assertion consumption endpoint, expected JWT/SAML payload for user identity) are TBD. (Inferred from D001)
Photo Upload API: An API endpoint (e.g., POST /api/albums/{albumId}/photos) to accept photo file uploads. Expected payload: multipart/form-data with image files and associated metadata. (Inferred from D006)

Modeling Start API: An API endpoint (e.g., POST /api/projects/{projectId}/model/start) to initiate the modeling process. Expected payload: project ID and (for V1) implicitly 'Normal Mode' parameters. (Inferred from D004, D019)
Modeling Progress Webhook/API: An interface for the modeling computation service to push progress updates and stage statuses to This System. (Inferred from D009, D010)
Photo Selection/Import API (Project): An API endpoint (e.g., POST /api/projects/{projectId}/photos/import) to link/copy selected photos to a project. Expected payload: project ID and a list of photo IDs. (Inferred from D011)
