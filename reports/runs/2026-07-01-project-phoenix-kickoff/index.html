Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a new user navigates to the registration page, when they provide a valid Email, Full Name, Role, and Company Name, then they shall be successfully registered and authenticated.
Given a user is on the login page, when they provide valid email and password credentials, then they shall be successfully authenticated and redirected to the project dashboard.
Given a user is on the login page, when they click "Sign in with Google", then they shall be authenticated via Firebase (Google Sign-in) and redirected to the project dashboard.
Given a user has an active session, when 24 hours pass since their last interaction, then their session token shall expire, requiring re-authentication upon next action.
Given a new user attempts to register, when they submit the registration form with a valid Email, then the system shall accept it as a required field.
Given a new user attempts to register, when they submit the registration form with a valid Full Name, then the system shall accept it as a required field.
Given a new user attempts to register, when they submit the registration form with a valid Role, then the system shall accept it as a required field.
Given a new user attempts to register, when they submit the registration form with a valid Company Name, then the system shall accept it as a required field.
Given a new user attempts to register, when they submit the registration form including an address, then the system shall ignore or not store the address.
Given a new user attempts to register, when they submit the registration form including a telephone number, then the system shall ignore or not store the telephone number.
Given an authenticated user navigates to the project dashboard, then it shall display active user metrics.
Given an authenticated user navigates to the project dashboard, then it shall display the system status.
Given an authenticated user navigates to the project dashboard, then it shall display a list of projects.
Given an attempt to connect to the PostgreSQL database instance from the public internet, then the connection shall be rejected and the instance shall not be publicly exposed.
Given the Cloud Run backend application, when it attempts to connect to the PostgreSQL database, then it shall establish the connection exclusively via Serverless VPC Access.
Test Scenarios
Verify successful user registration using email, full name, role, and company name.
Verify successful user login using valid email and password credentials.
Verify successful user login using Google Sign-in.
Verify the project dashboard loads correctly after successful authentication, displaying active user metrics, system status, and project lists.
Verify session token expires after 24 hours of inactivity and requires re-authentication.
Verify that attempting to register without providing an email address results in an error message and prevents registration.
Verify that attempting to register without providing a full name results in an error message and prevents registration.
Verify that attempting to register without providing a role results in an error message and prevents registration.
Verify that attempting to register without providing a company name results in an error message and prevents registration.
Verify that providing an address or telephone number during registration does not result in these fields being stored or used.
Verify the PostgreSQL database instance is inaccessible from outside the private VPC.
Verify Cloud Run application establishes its database connection via Serverless VPC Access as intended.
Edge Cases
Attempt to register with an email address that is already in use.
Attempt to register with invalid email formats (e.g., missing '@', invalid domain).
Simultaneous login attempts from the same user account from different devices or browsers.
User performs actions immediately before the 24-hour session token expiration, then attempts another action just after.
User attempts to access a protected dashboard feature after their session has expired.
Dashboard displays data correctly when there are no active users, no projects, or the system status is critical/error.
Testing performance under high load conditions to assess if the system remains 'super fast'.
Attempting to register and then immediately logging out and logging back in.
Unresolved Behaviours
The definition of "super fast" for system performance is subjective; quantifiable metrics (e.g., specific response times for key actions, page load times) are needed.
The specific acceptance criteria for the overall user registration flow beyond field collection (e.g., email verification, account activation, post-registration onboarding) are not fully detailed.
The detailed features, specific data points, update frequency, and interactivity of 'active user metrics', 'system status', and 'project lists' on the dashboard need to be specified in the next sync meeting.
The specific behaviors for new user registration via Google Sign-in (e.g., how Full Name, Role, Company Name are collected if not provided by Google profile or if they are mandatory post-Google sign-in) are unclear.
The scope and impact of the 'Role' field during user registration are not defined (e.g., does it imply different permissions, access levels, or dashboard views?).
