Acceptance Criteria
Given the system is launched, When a user attempts to access the application, Then an authentication mechanism is presented.
Given a user successfully authenticates, When they navigate to the primary application page, Then a functional project dashboard is displayed.
Given Firebase Authentication is integrated, When a a user attempts to sign in via Google, Then the system successfully authenticates the user using their Google account.
Given Firebase Authentication is integrated, When a user attempts to sign in via email, Then the system successfully authenticates the user using their email credentials.
Given a new user is registering, When they provide their email address, Then the system collects and stores their email address.
Given a new user is registering, When they provide their full name, Then the system collects and stores their full name.
Given a new user is registering, When they provide their role, Then the system collects and stores their role.
Given a new user is registering, When they provide their company name, Then the system collects and stores their company name.
Given a new user is registering, When the 'Company Name' field is left blank, Then the system prevents registration and displays a validation error indicating it is a required field.
Given a user is successfully logged in, When viewing the project dashboard, Then the dashboard displays current active user metrics.
Given a user is successfully logged in, When viewing the project dashboard, Then the dashboard displays the current system status.
Given a user is successfully logged in, When viewing the project dashboard, Then the dashboard displays a list of projects.
Given a user has an active session, When 24 hours have passed since the session token was issued, Then the user's session token is expired, requiring re-authentication upon next action.
Given the PostgreSQL database is deployed, When an attempt is made to connect from a public internet IP address, Then all connection attempts are blocked by network security configurations.
Test Scenarios
Verify successful user registration via email with valid Email, Full Name, Role, and Company Name. Confirm user can log in with these credentials.
Verify successful user login using valid email and password credentials. Confirm redirection to the dashboard.
Verify successful user login using Google Sign-in. Confirm redirection to the dashboard.
Verify that attempting to register with an existing email address displays an appropriate error message.
Verify that attempting to log in with incorrect email/password displays an appropriate error message.
Verify that the dashboard loads correctly and displays placeholder or default data for 'Active User Metrics', 'System Status', and 'Project Lists'.
Verify that after a user logs in, their session remains active for at least 23 hours and 59 minutes.
Verify that after a user logs in, and remains idle for 24 hours and 1 minute, they are prompted to re-authenticate when attempting an action.
Edge Cases
Attempt user registration with invalid email formats (e.g., missing @, missing domain, special characters).
Attempt user registration with extremely long inputs for Full Name, Role, and Company Name fields (e.g., character limits).
Attempt user login with an account that has been disabled or locked (if such functionality exists within Firebase).
Test concurrent logins for the same user account from different devices or browsers.
Repeated rapid failed login attempts (e.g., brute-force scenarios) and observe Firebase's rate limiting/security measures.
Accessing protected application routes directly via URL without an active session or with an expired session.
Attempting to establish a direct network connection to the PostgreSQL database from outside the private VPC.
Monitor system responsiveness during typical user operations (login, dashboard load) under simulated moderate load to identify potential performance bottlenecks against the 'super fast' requirement.
