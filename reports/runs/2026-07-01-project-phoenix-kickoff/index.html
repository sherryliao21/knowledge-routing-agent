Quality Assurance

Acceptance criteria, test scenarios, edge cases, and untestable behaviours.

Acceptance Criteria
Given a user navigates to the registration page, When they provide a valid Email, Full Name, Role, and Company Name, Then they SHALL be able to successfully register.
Given a registered user, When they attempt to log in using their registered email and password, Then they SHALL be successfully authenticated and redirected to the project dashboard.
Given a user, When they attempt to log in using Google Sign-in, Then they SHALL be successfully authenticated and redirected to the project dashboard.
Given a user, When their session token has expired after 24 hours, Then they SHALL be prompted to log in again upon attempting to access protected resources.
The user registration process SHALL NOT require input fields for 'Address' or 'Telephone Number'.
Given a successfully logged-in user, When they access the project dashboard, Then the dashboard SHALL display active user metrics.
Given a successfully logged-in user, When they access the project dashboard, Then the dashboard SHALL display system status information.
Given a successfully logged-in user, When they access the project dashboard, Then the dashboard SHALL display a list of projects.
The system SHALL operate at a high performance level, perceived as 'super fast' by end-users.
The PostgreSQL database instance SHALL NOT be directly accessible from the public internet.
The Cloud Run backend application SHALL be able to establish a secure connection to the PostgreSQL database via Serverless VPC Access.
Test Scenarios
Verify a new user can successfully register using a valid email, full name, 'Admin' role, and 'Test Company' company name.
Verify a new user can successfully register using Google sign-in and their profile information is captured.
Verify attempting to register with one or more of Email, Full Name, Role, or Company Name fields empty results in an appropriate error message.
Verify attempting to register with an invalid email format (e.g., 'test@.com') results in an appropriate error message.
Verify a registered user can successfully log in with correct email and password.
Verify a registered user can successfully log in using Google sign-in.
Verify attempting to log in with incorrect email or password results in an appropriate error message.
Verify that after a user's session token expires (24 hours), they are logged out or prompted to re-authenticate when trying to access protected dashboard content.
Verify a logged-in user can navigate to and view the project dashboard.
Verify an unauthenticated user attempting to access the dashboard is redirected to the login/registration page.
Verify direct connection attempts to the PostgreSQL database from a public IP address are rejected.
Verify the Cloud Run backend successfully connects to and queries the private PostgreSQL database.
Edge Cases
Test user registration and login fields (Full Name, Role, Company Name, Email) with maximum allowed character lengths.
Test user registration and login fields with various special characters to ensure proper handling and prevent injection vulnerabilities.
Verify system behavior when a single user account logs in concurrently from multiple different devices or browsers.
Attempt to create a user account with an email address that already exists within the system.
Simulate high concurrent user load to evaluate if the system maintains 'super fast' performance and if PostgreSQL sessions become a performance blocker.
Attempt to access Cloud Run backend endpoints with malformed, expired, or missing authentication tokens.
Unresolved Behaviours
